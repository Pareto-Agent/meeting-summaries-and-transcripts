Tooling: Replit vs Claude Code
  • Replit: fast prototyping, spins up live site + database automatically, prefers to fill in gaps itself — hard to change direction once it’s decided on an approach
  • Claude Code: write the spec; easier to iterate and make changes; feeds into GitHub
  • Workflow: prototype in Replit to validate ideas quickly, parallel-path a spec in Claude Code for the real build
  • Replit and Claude Code are separate environments — no automatic link between them; can pass a spec from Claude Code into Replit manually
Claude Code Best Practices (Aaron’s Walkthrough)
  • Use terminal over desktop app where possible
  • /plan before running anything — avoids failed runs and wasted tokens
  • ultrathink / ultraplan for large initial builds only
  • Advisor mode: turns on Opus 4 to review Sonnet’s output after each run — catches errors, saves tokens net
  • Compact frequently (especially past ~300–400k context); use /compact to summarise prior steps
  • CLAUDE.md file: run /init to create it; keep it updated after every run with repo layout, constraints, and a handoff prompt for the next agent session
    • Add instruction inside CLAUDE.md: “after every run, update this file and write a handoff document”
  • Spin up research agents by including “spin up a research agent” in the prompt
  • Skills/MCPs worth exploring: browser tools (screenshots), repo cleanup, PRD/spec generation — large open-source library available online
MCP vs API
  • MCP: broad, schema-free access to a data source — more flexible for agents, but best-efforts and prone to hallucination; harder to audit; bigger security surface
  • API: requires knowing the exact fields/objects to query — more reliable and auditable, but slower to set up
